# Click Mini-Apps Platform — Предложение по программе для разработчиков

**Дата**: 11 мая 2026  
**Аудитория**: продуктовая и техническая команда Click  
**Документ**: краткая презентация (2 стр.)  
**Связанные файлы**: `02-architecture.md` (полное описание), `03-requirements.md` (требования)

---

## Цель

Дать **внешним разработчикам** (юр.лицам, ИП, самозанятым в Узбекистане) формализованный, безопасный и предсказуемый путь от идеи до production мини-аппа внутри Click SuperApp. Click при этом сохраняет:

- **Полный контроль** над тем, что попадает к пользователям (KYC, ревью, kill-switch)
- **Низкую операционную нагрузку** (self-hosted модель — runtime у разработчика)
- **Прозрачную аудиторскую цепочку** для регулятора и собственного compliance

Цель — превратить разработку мини-аппов из «дёрни знакомого в Click для интеграции» в публичную программу с понятными правилами, документами и SLA.

---

## Что предлагаем — в одном абзаце

Click открывает программу `miniapps.click.uz` с кабинетом разработчика, тестовой инфраструктурой (собственный GitLab + sandbox API + sandbox-эмулятор Click WebView), документацией и формализованным процессом ревью → approve → публикация в нативном Click marketplace. Мини-аппы хостятся на стороне разработчика; Click выступает identity provider'ом (через WebView Bridge с инжекцией `click_profile`), процессором платежей (HMAC-подписанный backend API), доставщиком push (через registry утверждённых шаблонов) и каталогом. Безопасность обеспечивается четырёхслойной защитой: domain pinning, daily fingerprint check, mandatory delta-review для significant changes, kill-switch при нарушениях. Эскалация при повторных нарушениях — всегда через менеджера, не автоматически.

## Архитектура — три плоскости

| Плоскость | Что внутри | Кто пользуется |
|---|---|---|
| **Developer Plane** | `miniapps.click.uz` (кабинет), `git.miniapps.click.uz` (GitLab CE), `docs.miniapps.click.uz` (документация) | разработчик |
| **Runtime Plane** | `sandbox.api.click.uz`, `api.click.uz`, `auth.click.uz`, нативный Click marketplace | мини-апп + пользователи Click |
| **Governance Plane** | Admin Console, Daily Fingerprint Bot, Push Template Registry, Audit Log, Moderation Queue | сотрудники Click |

Между плоскостями — только определённые API. Никаких ad-hoc обходов, всё под audit log.

---

## Что получает разработчик

1. Самостоятельная регистрация за 15 минут, сразу — sandbox без проверок
2. GitLab-репо из template'а под целевой capability-уровень (L1/L2/L3/L4)
3. Sandbox Launcher — эмулятор Click WebView прямо в кабинете (не нужно ставить Click для тестов)
4. 10–20 тестовых пользователей с разными KYC-статусами / атрибутами
5. Fake-платежи и fake-push в песочнице (с возможностью форсить отказы и таймауты)
6. **Cabinet Security Scanner** в кабинете (shift-left) — ровно те же проверки, что и на финальном ревью
7. Webhook Inspector — Click ловит все webhook'и, если у разработчика нет публичного URL
8. Документация на русском + узбекском, готовые SDK для JS/Node.js
9. Прозрачный submission flow со SLA 30 мин (auto) + 7 раб.дней (manual) + 3 раб.дня (resubmission delta)
10. Прозрачная политика жизненного цикла — что приводит к auto-disable, как этого избежать, как восстановиться

---

## Что получает Click

1. **Контролируемая экосистема партнёрских сервисов** в нативном marketplace без операционной нагрузки на хостинг
2. **Полный compliance trail** — KYC, договоры, ревью, audit log, fingerprint history
3. **Защита бренда** — kill-switch при нарушениях, эскалация повторных проблем на менеджеров (не на автомат)
4. **Подготовка к монетизации** — поле `fee_amount` в payment API живёт с нуля, активация когда захотим (на пилоте — бесплатно)
5. **Стандартизация** — все мини-аппы (включая текущие embedded) могут переехать на единый протокол
6. **Регуляторная готовность** — explicit consent flows, persistence ПДн в Узбекистане, отчётность по каждой транзакции и каждому push
7. **Привлечение качественных разработчиков** — pen-test от утверждённых пентестеров отсеивает шарашек на входе

## Capability levels (что мини-аппы могут делать)

| Уровень | Возможности | KYC tier разработчика |
|---|---|---|
| **L1** | SSO — узнать user_id, имя, телефон, аватар | K1 (юр.лицо/ИП/самозанятый) |
| **L2** | + расширенный профиль (KYC статус, регион, язык) | K1 |
| **L3** | + инициация платежей со счёта Click + webhooks | K2 (K1 + расчётный счёт + договор) |
| **L4** | + push по утверждённым шаблонам с окнами/лимитами | K3 (K2 + DPO + privacy policy + одобренные шаблоны) |

Push в L4 — **только по шаблонам**, утверждённым Click заранее. Свободного текста нет в принципе. У каждого шаблона — окно отправки и rate-limit (per user, per day).

---

## Безопасность как ключевая ценность

Четыре уровня защиты:

1. **Domain pinning** — мини-апп зафиксирован на конкретном домене (DNS TXT-record), переезд = новый submission
2. **Daily fingerprint** — бот каждые 24 ч сравнивает мини-апп с эталоном на момент approve, критические расхождения → auto-disable
3. **Mandatory delta-review** — любое значимое изменение (новые scopes, новые внешние домены, новые формы) проходит через кабинет
4. **Kill-switch + reporting** — пользователь Click может пожаловаться, модератор может мгновенно отключить мини-апп

Дополнительно для L3/L4 — обязательный pen-test от утверждённого Click-пентестера, валиден 12 мес. **Cabinet Security Scanner** интегрирован в кабинет с самого начала: разработчик видит проблемы на этапе разработки, а не на ревью.

## Что нужно от Click для запуска программы

**Технически:**
1. Поднять GitLab CE на инфре Click
2. Создать sandbox-стенд (поддомены, тестовые пользователи, fake-платежи/push, Webhook Inspector)
3. Расширить существующий Click API: WebView Bridge с JWT-выдачей, HMAC-канал для backend, scope-система
4. Push Template Registry + rate-limit/окна enforcement
5. Daily Fingerprint Bot (headless Chromium + cron)
6. Admin Console для менеджеров

**Юридически:**
1. Шаблоны договоров оферты для K1, K2, K3
2. Политика контента (что запрещено в Click marketplace)
3. Список утверждённых пен-тестеров
4. Регламент эскалации (kill-switch, расторжение договора, передача в фин.разведку)

**Организационно:**
1. Команда модераторов (review, complaints handling)
2. Команда compliance (KYC, договоры)
3. Security Officer (фрод, утечки)
4. Документация на 3 языках (RU primary, UZ latin + cyrillic, EN опционально)

Решение по фазированию запуска (что в каком порядке — L1 первым? L3 вместе с L1?) — за Click team.
